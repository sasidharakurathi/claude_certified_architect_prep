Per the official guide: the exam validates that a practitioner can "make informed decisions about tradeoffs when implementing real-world solutions with Claude." Target candidate: a solution architect with 6+ months of hands-on experience with Claude APIs, the Agent SDK, Claude Code, and MCP.

Domain 1

Agentic Architecture & Orchestration

27% of the exam — the single biggest domain
Learn more: Subagents doc ↗ Hooks doc ↗ Full resources & practicals →

7 official Task Statements, from the agentic loop through session management.

1.1 Design and implement agentic loops

The loop: send a request → inspect stop_reason → if "tool_use", execute and append results to history, then repeat → if "end_turn", stop. This is model-driven decision-making, not a pre-configured decision tree.

Named anti-patterns

Parsing natural-language signals to decide when to stop; an arbitrary iteration cap as the primary stop mechanism; checking for assistant text content as a "done" indicator. None of these are the real signal — use stop_reason.

1.2 Orchestrate multi-agent systems: coordinator-subagent patterns

Hub-and-spoke: the coordinator owns all inter-subagent communication, error handling, and routing. Subagents run with isolated context — they don't automatically inherit the coordinator's history. The coordinator decomposes, delegates, aggregates, and decides which subagents to invoke based on query complexity.

Named anti-pattern — official sample question

Overly narrow task decomposition. A coordinator researching "AI's impact on creative industries" that only spins up subtasks for digital art, graphic design, and photography misses music, writing, and film — even though every subagent executes its assigned task perfectly. The failure is in what the coordinator assigned, not how subagents performed.

Skills: dynamically selecting which subagents to invoke (not always the full pipeline); partitioning scope to avoid duplication; iterative refinement loops (evaluate synthesis for gaps → re-delegate with targeted queries → re-invoke); routing all communication through the coordinator for observability.

1.3 Configure subagent invocation, context passing, and spawning

⚠️ The mechanism is the Task tool — a coordinator's allowedTools must include "Task" to invoke subagents. Subagent context is not automatically inherited — include what's needed explicitly in its prompt. AgentDefinition configures description, system prompt, and tool restrictions per subagent type.

  • Use structured data formats to separate content from metadata (source URLs, document names, page numbers) when passing context, to preserve attribution
  • Spawn parallel subagents by emitting multiple Task calls in a single coordinator response, not across separate turns
  • Design coordinator prompts around goals and quality criteria, not step-by-step procedures, so subagents can adapt

1.4 Multi-step workflows: enforcement and handoff

Key phrase, memorize it

When deterministic compliance is required, "prompt instructions alone have a non-zero failure rate." That's the guide's own reasoning for choosing programmatic prerequisites over prompts.

Example: block process_refund until get_customer has returned a verified ID. For multi-concern requests: decompose into distinct items, investigate in parallel with shared context, then synthesize one response. Structured handoff summaries to humans should include customer ID, root cause, refund amount, recommended action.

1.5 Agent SDK hooks: tool call interception and data normalization

  • PostToolUse — normalize heterogeneous data formats from different MCP tools (Unix timestamps vs. ISO 8601 vs. numeric codes) before Claude reasons over them
  • Tool-call interception — block a policy-violating action (e.g., a refund over $500) and redirect to human escalation

Choose hooks over prompting whenever a business rule needs a guaranteed, not probabilistic, outcome.

1.6 Task decomposition strategies

Prompt chaining (fixed sequential pipeline) for predictable multi-aspect reviews — e.g., analyze each file, then one cross-file integration pass. Dynamic/adaptive decomposition for open-ended investigation — generate subtasks based on what's discovered.

Named anti-pattern — official sample question

A single review pass over 14 files produces inconsistent depth and even contradictory findings. The fix is per-file passes + a separate cross-file integration pass — not a bigger context window (doesn't fix attention dilution) and not requiring majority-consensus across repeated full-context runs (suppresses real intermittent findings).

1.7 Session state, resumption, and forking

⚠️ --resume <session-name> continues a specific named session. fork_session branches an independent copy from a shared baseline to explore a divergent approach. When resuming after code changes, explicitly tell the agent which files changed — it won't auto-detect drift. If prior tool results are stale, starting fresh with an injected structured summary beats resuming.

Quick self-check

  • I can state the loop's stop_reason rule and the three termination anti-patterns.
  • I know the Task tool is the subagent mechanism and allowedTools must include "Task".
  • I can explain fork_session vs. --resume <session-name>.
  • I can recognize "overly narrow coordinator decomposition" as a root cause distinct from blaming a working subagent.
  • I know why per-file + integration passes beat one big pass, and why a bigger window doesn't fix that.
  • I know when hooks (not prompts) are required for guaranteed compliance.

Domain 2

Tool Design & MCP Integration

18% of the exam
Learn more: MCP Architecture ↗ Define tools ↗ Full resources & practicals →

5 official Task Statements, from tool description design through built-in tool selection.

2.1 Design effective tool interfaces

Tool descriptions are the primary mechanism models use for tool selection. Minimal descriptions ("Retrieves customer information") cause unreliable selection among similar tools.

Official sample question — confirmed judgment call

When misrouting is caused by thin descriptions, the correct first step is expanding the descriptions (input formats, example queries, edge cases, boundaries) — not few-shot examples (token overhead, doesn't fix root cause), not a routing/classifier layer (over-engineered), and not immediately consolidating tools (valid, but more than a "first step").

Also watch for keyword-sensitive system-prompt wording creating unintended tool associations that override otherwise good descriptions.

2.2 Structured error responses for MCP tools

The MCP isError flag signals failure. Categorize: transient / validation / business / permission. Recommended fields: errorCategory, isRetryable, human-readable description; for business violations, retriable: false plus a customer-friendly explanation.

Two named anti-patterns

Generic errors ("Operation failed") hide context the agent needs to recover. And conflating an access failure (needs a retry decision) with a valid empty result (successful query, nothing found) breaks downstream reasoning either way.

Implement local recovery within subagents for transient failures; propagate to the coordinator only what couldn't be resolved locally, with partial results and what was attempted.

2.3 Distribute tools across agents; configure tool choice

Specific numbers from the guide

Giving an agent 18 tools instead of 4–5 degrades selection reliability by increasing decision complexity. Agents with tools outside their specialization tend to misuse them.

tool_choice: "auto" (model decides) · "any" (must call a tool, chooses which) · forced {"type":"tool","name":"..."} (must call this specific tool first). Give narrow cross-role tools only for specific high-frequency needs (e.g., a scoped verify_fact tool), routing complex cases through the coordinator.

2.4 Integrate MCP servers into Claude Code and agent workflows

ScopeLocationUse
Project.mcp.jsonShared team tooling, version-controlled, ${ENV_VAR} expansion for credentials
User~/.claude.jsonPersonal/experimental servers

Tools from all configured servers are discovered at connection time and available simultaneously. MCP resources expose content catalogs (issue summaries, doc hierarchies, DB schemas) to cut down exploratory tool calls. Prefer existing community servers (e.g., Jira) over custom builds for standard integrations.

2.5 Built-in tools: Read, Write, Edit, Bash, Grep, Glob

ToolUse for
GrepContent search — function names, error messages, imports
GlobFile path pattern matching (**/*.test.tsx)
Read / WriteFull file operations
EditTargeted edit via unique text match — falls back to Read+Write if the anchor isn't unique

Build codebase understanding incrementally: Grep for entry points, then Read to trace imports/flow — don't read everything upfront.

Quick self-check

  • I know why "expand the description" beats few-shot/routing/consolidation as the first fix for misrouting.
  • I can write a structured MCP error distinguishing the 4 error categories, and I know access-failure ≠ valid-empty-result.
  • I know the "18 vs 4–5" framing and all three tool_choice modes.
  • I know .mcp.json (project) vs. ~/.claude.json (user) MCP scoping.
  • I can pick the right built-in tool for a described scenario, including the Edit→Read+Write fallback.

Domain 3

Claude Code Configuration & Workflows

20% of the exam
Learn more: CLAUDE.md doc ↗ Claude Code overview ↗ Full resources & practicals →

6 official Task Statements, from CLAUDE.md hierarchy through CI/CD integration.

3.1 CLAUDE.md hierarchy, scoping, modular organization

LevelLocationShared?
User~/.claude/CLAUDE.mdNo — never propagates via VCS
Project.claude/CLAUDE.md or root CLAUDE.mdYes, via version control
DirectorySubdirectory CLAUDE.mdYes, scoped to that subtree
Diagnostic pattern

A new teammate not receiving expected instructions is the classic symptom of instructions living at user-level instead of project-level.

@import keeps CLAUDE.md modular. .claude/rules/ splits into topic files as an alternative to one monolith. /memory verifies which files are actually loaded — use it to diagnose inconsistent behavior.

3.2 Custom slash commands and skills

Commands: project-scoped .claude/commands/ (shared, VCS) vs. user-scoped ~/.claude/commands/ (personal). Skills: .claude/skills/ with SKILL.md frontmatter — context: fork (isolated sub-agent context, keeps verbose output out of the main conversation), allowed-tools (restrict tools during execution), argument-hint (prompt for required params). Personal variants go in ~/.claude/skills/ under a different name.

3.3 Path-specific rules for conditional convention loading

.claude/rules/ with YAML frontmatter paths: glob patterns — loads only when Claude touches a matching file.

Official sample question

Test files scattered throughout a codebase (e.g., Button.test.tsx next to Button.tsx) need conventions that apply by file type, regardless of directory. A glob like paths: ["**/*.test.tsx"] handles this; subdirectory CLAUDE.md files can't (they're directory-bound), and consolidating into root CLAUDE.md relies on unreliable inference.

3.4 Plan mode vs. direct execution

Plan mode: large-scale changes, multiple valid approaches, architectural decisions, multi-file modifications — explore and design before committing. Direct execution: simple, well-scoped, single approach. The Explore subagent isolates verbose discovery output to prevent context-window exhaustion.

Official sample question

For a monolith→microservices restructuring (dozens of files, architectural decisions) — plan mode is correct from the start. "Start direct and switch to plan mode if surprised" is wrong because the complexity was already known upfront, not something that might emerge later.

3.5 Iterative refinement

2–3 concrete input/output examples when prose descriptions get inconsistent results. Test-driven iteration: write tests first, iterate on failures. The interview pattern: have Claude ask clarifying questions before implementing in an unfamiliar domain. Batch interacting issues into one message; fix independent issues sequentially.

3.6 Integrate Claude Code into CI/CD pipelines

-p (or --print) for non-interactive mode — prevents hangs waiting for input. --output-format json + --json-schema for machine-parseable findings (auto-posted as inline PR comments). CLAUDE.md carries project context (testing standards, fixtures, review criteria) to CI-invoked runs.

Session isolation

The same session that generated code is less effective reviewing its own changes than an independent instance — it retains its own generation reasoning. (Connects to Domain 4's multi-instance review content.)

Quick self-check

  • I can diagnose the "new teammate missing instructions" symptom against the CLAUDE.md hierarchy.
  • I know context: fork, allowed-tools, argument-hint as skill frontmatter fields.
  • I know why glob-pattern rules beat subdirectory CLAUDE.md for file-type-based conventions.
  • I can pick plan mode vs. direct execution based on what's already known, not what might emerge.
  • I know -p, --output-format json, --json-schema, and why self-review is weaker than independent review.

Domain 4

Prompt Engineering & Structured Output

20% of the exam
Learn more: Prompting best practices ↗ GitHub tutorial ↗ Full resources & practicals →

6 official Task Statements. Correction from earlier drafts: the exam's actual model of "reliable structured output" is tool_use + JSON schema — there is no separate constrained-decoding "Structured Outputs" feature tested, and prompt caching is out of scope.

4.1 Explicit criteria to improve precision, reduce false positives

Explicit categorical criteria beat vague instructions — "flag comments only when claimed behavior contradicts actual code" beats "check that comments are accurate."

Named anti-pattern

"Be conservative" or "only report high-confidence findings" fail to improve precision compared to specific criteria — confidence filtering isn't a substitute for defining what to report vs. skip.

A high false-positive rate in one category undermines trust in all categories — temporarily disabling a noisy category while fixing its prompt is a legitimate interim fix.

4.2 Few-shot prompting for consistency and quality

Few-shot examples are the most effective technique for consistent output when prose alone is inconsistent. Good for: ambiguous-case handling, generalization to novel patterns, and reducing hallucination in extraction.

2–4 targeted examples showing the reasoning for choosing one action over plausible alternatives; examples demonstrating the desired output format; examples distinguishing acceptable patterns from genuine issues; examples covering varied document structures.

4.3 Enforce structured output using tool use and JSON schemas

The exam's actual answer for "reliable structured output"

tool_use with a JSON schema — eliminates JSON syntax errors by construction. It does not prevent semantic errors (line items not summing to total, values in the wrong field).

Same tool_choice modes as Domain 2, applied here: "any" when the document type is unknown and multiple extraction schemas exist; forced named tool to guarantee a specific extraction runs first. Schema design: optional/nullable fields prevent fabricated values; enum + "other" + detail-string pattern for extensibility; an explicit "unclear" value for genuine ambiguity.

4.4 Validation, retry, and feedback loops

Retry-with-error-feedback: include the original document, the failed extraction, and the specific validation error.

The limit of retries

Retries fix format/structural errors but are ineffective when the required information is simply absent from the source — no amount of retrying recovers information that was never there.

detected_pattern fields let you analyze which constructs trigger false positives. Self-correction flows: extract calculated_total alongside stated_total to flag discrepancies; add a conflict_detected boolean.

4.5 Batch processing strategies

Message Batches APIFact
Cost~50% savings
LatencyUp to 24h, no guaranteed SLA
Tool callingNo multi-turn tool calling within a single request
Correlationcustom_id per request/response pair
Official sample question

Batch is right for overnight/weekly non-blocking work; wrong for a blocking pre-merge check a developer is waiting on. Resubmit only failed documents by custom_id; refine the prompt on a small sample before batch-processing a large volume.

4.6 Multi-instance and multi-pass review architectures

A session that generated code retains its own reasoning, making it less likely to question its own decisions — even with self-review instructions. An independent review instance catches subtle issues more reliably. Split large reviews into per-file + cross-file integration passes (see Domain 1, TS 1.6). Have the model self-report a confidence score per finding to enable calibrated review routing.

Quick self-check

  • I know tool_use + JSON schema is the exam's actual structured-output answer — not a separate feature.
  • I can explain syntax vs. semantic errors and give the "totals don't sum" example.
  • I know when retries help (format) vs. can't help (info absent from source).
  • I know the Batch API's exact tradeoffs, including "no multi-turn tool calling."
  • I know why independent review beats self-review.

Domain 5

Context Management & Reliability

15% of the exam
Learn more: Context windows doc ↗ Subagents doc ↗ Full resources & practicals →

6 official Task Statements. Correction from earlier drafts: prompt-caching mechanics (TTL, breakpoints, pricing) are out of scope — this domain is about information management and reliability judgment, not caching internals.

5.1 Manage conversation context across long interactions

Progressive summarization risks condensing numbers, dates, and customer-stated expectations into vague summaries. "Lost in the middle": models process the beginning/end of long inputs reliably but may omit middle sections. Tool results accumulate disproportionately to relevance (40+ fields when only 5 matter).

Fixes: a persistent "case facts" block outside summarized history; trim verbose tool outputs before they accumulate; place key findings at the beginning of aggregated inputs with explicit headers; have upstream agents return structured data instead of verbose prose when downstream context budgets are tight.

5.2 Escalation and ambiguity resolution

Two named anti-patterns — official sample question

Sentiment-based escalation and self-reported confidence scores are both unreliable proxies for actual case complexity.

Correct triggers: explicit customer request for a human (honor immediately, no investigation first), genuine policy gap/exception, inability to progress. If merely frustrated but the issue is solvable, acknowledge and offer to resolve — escalate only if they reiterate. Multiple ambiguous matches → ask for an additional identifier, never guess heuristically.

5.3 Error propagation across multi-agent systems

Two opposite anti-patterns

Silently suppressing an error (empty result marked success) and terminating the whole workflow on one failure are both wrong — both remove the coordinator's ability to make an informed recovery decision.

Structured error context (failure type, attempted query, partial results, alternatives) enables recovery. Distinguish an access failure (needs retry decision) from a valid empty result (doesn't).

5.4 Context management in large codebase exploration

Context degradation shows up as inconsistent answers referencing "typical patterns" instead of specific classes found earlier. Fixes: subagent delegation to isolate verbose exploration; scratchpad files for key findings; summarize before the next phase rather than carrying raw exploration forward; manifests for crash recovery (each agent exports state, coordinator reloads on resume); /compact once verbose discovery fills the window.

5.5 Human review workflows and confidence calibration

The masking risk

A 97% aggregate accuracy figure can hide much worse performance on a specific document type or field.

Stratified random sampling of high-confidence extractions specifically catches novel error patterns a naive low-confidence-only check would miss. Validate by document type and field segment before scaling back review; route low-confidence/contradictory cases to scarce reviewer capacity.

5.6 Information provenance and uncertainty in synthesis

Source attribution is lost when findings are compressed without preserving claim-source mappings. Require subagents to output structured mappings (source URL, document name, excerpt) that synthesis preserves. Conflicting statistics from credible sources → include both with attribution, don't arbitrarily pick one. Require publication/collection dates so temporal differences aren't misread as contradictions.

Quick self-check

  • I can explain "lost in the middle" and name two mitigations.
  • I know sentiment-based escalation and self-reported confidence are both named anti-patterns.
  • I know why suppressing errors and terminating the whole workflow are opposite but equally wrong.
  • I know scratchpad files, phase summarization, and crash-recovery manifests for context degradation.
  • I know why a 97% aggregate figure can hide a bad segment.
  • I know why conflicting source data should be preserved with attribution, not resolved arbitrarily.
